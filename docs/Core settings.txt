# Core Settings Integration Guide

1. Doctor Preferences Integration

(a) Integrate **DOCTOR_PREFERENCES_API_GUIDE** within **Core Settings**.
(b) Apply the same implementation logic already used in:
(i) Cases
(ii) Follow-ups



2. Currency Symbol Update

(a) The **₹ symbol is not provided by the backend**.
(b) Replace all occurrences of **₹** with **“Rs”** across the application (UI level).




4. Dynamic Enum Section (Core Settings)

Create a **Dynamic Enum Management** module in Core Settings.

(a) Follow **ENUM_PREFERENCES_FRONTEND_GUIDE** for UI and behavior.
(b) Apply dynamic enum usage across the entire application (all dropdowns).

API Endpoint

**GET** `/enums/doctor/all`
Returns: List of all enum groups with their options.



# Master Enum Overview

The system currently contains the following enum groups:

(a) AppointmentStatus
(b) CaseStatus
(c) ConsultationType
(d) DayOfWeek
(e) ExceptionType
(f) FollowupStatus
(g) ManufacturerEnum
(h) FormEnum
(i) PatientGender
(j) PrescriptionStatus
(k) PrescriptionType
(l) RepetitionEnum
(m) ScaleEnum

Enum Record Structure

Each enum item contains:

(a) id (UUID)
(b) value (system key)
(c) label (display text)
(d) sort_order
(e) is_active
(f) is_system
(g) enum_type
(h) created metadata

Implementation Notes

(a) All dropdowns in the application must consume data from `/enums/doctor/all`.
(b) Filter options using:
(i) enum_type
(ii) is_active = true
(iii) sort_order for display sequence

(c) Dynamic Enum module should allow:
(i) Add new options
(ii) Enable/disable options
(iii) Maintain system vs custom values
